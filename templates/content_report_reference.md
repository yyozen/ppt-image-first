# content_report.md

Use this file as the pre-style content basis when the user has not already provided a complete report-like narrative.

## Purpose

This file should be created after `需求确认` and before style-boundary alignment.
Its job is to turn the user's topic, scattered notes, or partial material into a small research-style report / report-like article that is strong enough to support:
- later `首页 / 目录页 / 正文页` previews
- `design_spec.md`
- `slide_blueprint.md`
- `spec_lock.md`

This file is not a short outline.
It should read like a compact but substantive upstream content report.

## Writing mode

Choose one of 2 modes:
- `整理但不扩写` — when the user already provided a strong complete report-like source
- `补强并报告化生成` — when the user only provided a topic, thin material, partial material, or scattered notes

Even in extraction mode, do not reduce the file to a bare bullet list.
Write a connected report-like text.

## Required sections

### I. Source Status
- source_level: `topic_only` / `thin_materials` / `partial_materials` / `scattered_materials` / `complete_report`
- user_provided_materials
- inferred_context
- source_limitations
- claims_policy

### II. Content Thesis
- what this deck is really about
- why this topic matters for the current audience
- what the deck is trying to prove, explain, defend, or persuade

### III. Narrative Body
Write this as connected prose, not just bullets.
It should usually include:
- background / topic framing
- core problem or core objective
- analysis body
- method / logic / solution / project path when applicable
- findings / value / implications when applicable
- conclusion direction suitable for PPT narration

If the user provided thin materials, strengthen this part enough that the PPT will not become shallow.
If the user provided complete materials, condense and organize them without inventing a new thesis.

### IV. Section Candidates
- likely chapter structure
- candidate section titles
- likely page clusters
- what each cluster needs to communicate

### V. Page Content Candidates
- cover framing candidate
- TOC structure candidate
- body-page candidate arguments
- key conclusion page candidate
- pages that may need comparison / process / diagram / evidence / case logic

### VI. Visualizable Content
- metrics or number types that would make sense if supported
- process or framework candidates
- comparison candidates
- relationship / hierarchy / flow candidates
- scenes, objects, entities, or anchors that can be visualized

### VII. Claim Status / Open Questions
Separate:
- user-provided facts
- inferred but usable claims
- claims that need confirmation before being treated as strong conclusions
- open questions that should not block style exploration

## Rules

- Write a small research-style report / report-like article, not a raw dump and not a minimal outline.
- Keep it compact enough to use, but deep enough to prevent shallow PPT planning.
- Prefer connected reasoning over generic bullet accumulation.
- Distinguish what came from the user vs what was inferred.
- Do not invent precise statistics, experiment results, citations, rankings, or institutional conclusions without support.
- If no reliable precise evidence exists, keep the claim general rather than falsely specific.
- The later preview set should draw from this file directly.
- The later 3 planning files should reference this file instead of rebuilding content from scratch.
